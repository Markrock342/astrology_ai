## 📌 สรุปผู้บริหาร

### ทำได้ดีแค่ไหน
พูดตรง ๆ: คุณสร้างมาเยอะและแกนกลางที่ยากที่สุดทำได้ "ดีจริง" — credit wallet + ledger เป็นระดับ production (atomic decrement + optimistic version lock, immutable CreditTransaction ใน transaction เดียว, idempotency key ที่ DB, และกฎทองของระบบคิดเงิน "AI ล้มเหลว = ไม่หักเครดิต" ทำถูกต้อง ซึ่งเป็นสิ่งที่คนพลาดกันมากที่สุด). prompt CMS, KnowledgeDoc, chart engine ก็ของจริงไม่ใช่ theater. แต่คะแนนรวมอยู่ที่ 5/10 เพราะช่องว่างไม่ได้อยู่ที่ "โค้ดเขียนไม่เป็น" — มันอยู่ที่ (1) ระบบที่ deploy จริงตอนนี้ยังใช้ซื้อขายไม่จบ 1 รอบ (email เป็น sandbox sender, VAPID ไม่ได้ตั้ง, และ admin password ดีฟอลต์ "ChangeMe123!" ถูก commit ขึ้น repo), (2) ครึ่งที่คุณมั่นใจที่สุด — "user เห็น usage เทียบเท่า ChatGPT" — คือครึ่งที่ยังไม่ได้สร้างจริง, และ (3) โมเดลข้อมูลของ payment (amount เป็น free-text, hardcode PRO, ไม่มี packageId, ไม่มีต้นทุน) ยังบางเกินกว่าจะรับเงินคนแปลกหน้า. สรุป: bones ดีมาก, แต่ perimeter รอบ ๆ เงินจริงยังไม่แน่น. งาน 2 วันที่โฟกัสถูกจุดดันจาก 5 ไป 7 ได้.

### พร้อม production ยัง
ยัง — อย่าเพิ่งเปิดรับเงินจริงสัปดาห์นี้แบบ self-serve. แต่ "yes-with-these-fixes" ถ้าแก้ config 4 อย่างก่อน: (1) หมุนรหัส SUPER_ADMIN บน production ทันที — ตอนนี้ admin@horasard.local / ChangeMe123! อยู่ใน .env.example ที่ commit ขึ้น repo แล้ว ใครก็ล็อกอินเข้า /admin/payments อนุมัติสลิปตัวเอง แจกเครดิตให้ตัวเองไม่จำกัด และอ่านสลิป+ข้อมูลวันเกิดลูกค้าทุกคนได้ (prisma/seed.ts:226). อันนี้พังก่อนเพื่อน และแพงที่สุด. (2) verify domain จริงใน Resend แล้วตั้ง EMAIL_FROM — ตอนนี้อนุมัติเงิน ฿199 แล้วอีเมลหายเข้า sandbox ลูกค้าไม่มีทางรู้ว่าได้รับอนุมัติ และ password reset ก็ตายด้วย (payment-notify.ts:88, mailer.ts:29). (3) ตั้ง Upstash Redis — ไม่งั้น rate limit ทุกตัวเป็นแค่ per-lambda memory ที่ fail-open ทำให้ brute-force รหัส admin ที่ published ยิ่งง่าย (rate-limit.ts:81). (4) ปิด double-approve race (payment-service.ts:141). ถ้าไม่แก้ 4 ข้อนี้ สิ่งที่พังก่อนคือ: ลูกค้าจริงคนแรกโอนเงินแล้วไม่มีอะไรบอกว่าผ่าน → เปิด ticket / chargeback, และ admin ของคุณถูกยึดได้จาก repo สาธารณะ. ทำได้ในระดับ MVP ที่ admin เฝ้าเอง สำหรับการซื้อครั้งแรก — แต่ยังไม่ใช่ระบบที่ปล่อยให้คนแปลกหน้าจ่ายซ้ำเองได้.

### เทียบเท่า ChatGPT จริงไหม
คำอ้าง "เทียบเท่า ChatGPT" จริงครึ่งหนึ่ง เท็จครึ่งหนึ่ง และครึ่งที่เท็จคือครึ่งที่เสียเงิน. ที่จริง: (1) token เป็นตัวเลขจริงจาก provider ไม่ใช่ chars/4 — เชื่อถือได้จริง. (2) admin ตั้ง maxOutputTokens/temperature/timeout ได้จริง อ่านสดจาก DB ทุก request ไม่ต้อง deploy — ไล่ code end-to-end แล้วเป็นจริงตามที่คุณอ้าง. (3) กฎ metering ที่สำคัญที่สุด "AI ล้มเหลว=ไม่หักเงิน" + idempotency + atomic charge ทำถูกต้อง ระดับที่ ChatGPT ก็ทำแบบนี้. ตรงนี้ให้เครดิตเต็ม. แต่ที่ไม่จริง: (a) user เห็นแค่ตัวเลขเครดิตเปล่า ๆ บน /account เท่านั้น — ไม่มี usage history, ไม่มี "3/20 วันนี้", ไม่มี ledger, ไม่เห็น balance ตอน chat, ไม่รู้ราคาก่อนกดส่ง. ChatGPT แสดง plan+usage+limit+billing ครบ. การที่ user รู้จัก quota ครั้งแรกผ่าน error แดง ไม่ใช่ "เทียบเท่า ChatGPT". (b) ไม่มี streaming เลย — ทุก reading เป็น blocking POST หลัง 30s timeout, user จ้อง spinner 10-30s. นี่คือช่องว่างใหญ่สุด. (c) admin เห็นต้นทุนเป็นศูนย์ — cost column เป็น NULL ตลอด, ตอบไม่ได้ว่า "เดือนนี้จ่าย Gemini เท่าไหร่" ต้องรอ statement บัตรเครดิต ขณะที่ ops console แบบ OpenAI แสดง spend ถึงเศษสตางค์ + hard budget cap + auto-cutoff. สรุป: ledger ซื่อสัตย์เรื่องที่มันบันทึก แต่มันไม่บันทึกเงิน (cost), บังคับ limit บน honor system ภายใต้ concurrency, และแสดงลูกค้าแทบไม่มีอะไร. เก่งเรื่องความถูกต้องของ transaction, ยังไม่ถึงเรื่อง visibility และ streaming.

### 🎯 จุดเปลี่ยนที่คุ้มที่สุด (ทำอันเดียวปิดหลายปัญหา)
เพิ่ม Payment.packageId FK แล้วให้ grant derive ทุกอย่างจาก package row ที่ซื้อ — แทนที่ amount free-text + hardcode packageCode:"PRO" + expiresAt:null. การเปลี่ยน schema-plus-one-function เดียวนี้ปิด money defect ใหญ่ที่สุดหลายอันพร้อมกัน: ทำให้ทุก payment มี audit link ว่าซื้ออะไรจริง (ปิดปัญหาไม่มี paper trail/ใบเสร็จ), ให้หน้า review เทียบ amount กับ Package.price ได้ (ปิดปัญหา "จ่าย 199 ได้ PRO ไม่ว่าจ่ายเท่าไหร่"), และทำให้ expiresAt=now+package.durationDays เป็นส่วนธรรมชาติของ grant (ปิด never-lapse). leverage สูงกว่า UI/rate-limit fix รายอันเพราะมันซ่อมโมเดลข้อมูลที่ทั้ง money loop ขาดไป — เปลี่ยน payment flow จาก "admin เพ่งเลขแล้ว hardcode PRO" เป็น transaction ที่มีราคาสินค้าจริง. (หมายเหตุ: สำหรับ "เปิดรับเงินสัปดาห์นี้" งานที่ปลดล็อกเร็วสุดคือ config sprint — หมุนรหัส admin + verify email domain + ตั้ง Upstash/VAPID — เพราะมันคือสิ่งที่ทำให้ระบบที่ deploy แล้วใช้ซื้อขายจบได้จริง.)

## ✅ จุดแข็ง (ของที่ทำได้ดีจริง — อย่าไปรื้อ)

- Credit ledger เป็นส่วนที่แข็งแรงที่สุดของทั้งระบบ — deductCredits (credit-service.ts:38-74) precheck ยอด แล้ว atomic updateMany guard ด้วย version, throw เมื่อ lost-update, และเขียน CreditTransaction ที่ลบไม่ได้ใน transaction เดียวกัน. wallet กับ ledger แยกจากกันไม่ได้เลย. ตรวจแล้วไม่ใช่เดา.
- กฎทองของระบบคิดเงินทำถูกต้อง: AI ล้มเหลว/timeout = log usage เป็น FAILED/TIMEOUT แต่ "ไม่หักเครดิต" (reading-service.ts:194-207), และ deduct + reading + usage log commit ใน prisma.$transaction เดียว (reading-service.ts:210-251). บั๊กคลาสสิก charge-before-call ไม่มีในระบบนี้.
- Idempotency บังคับที่ DB จริง ไม่ใช่แค่ใน app: @@unique([userId, idempotencyKey]) บน horoscope_readings (schema.prisma:513) + early-return (reading-service.ts:67-79). double-click หรือ retry ด้วย key เดิม ไม่มีทาง double-charge, และ retry ฝั่ง client reuse key เดิม (chat-view.tsx:349).
- Token เป็นตัวเลขจริงจาก provider ไม่ใช่ chars/4: gemini.ts:131-134 อ่าน usageMetadata, openai.ts:105-108 อ่าน usage.prompt_tokens. ข้อมูล token เชื่อถือได้จริง.
- ปุ่ม maxOutputTokens ที่คุณอ้างว่า "admin ตั้ง token ได้" เป็นเรื่องจริง — ไล่ end-to-end แล้ว: admin แก้ (ai-configs-manager.tsx:561) → Zod bound 64..32768 → persist (schema.prisma:441) → อ่านสดจาก DB ทุก request ไม่มี cache (router.ts:67) → ลงใน body ของ provider จริง (openai.ts:69, gemini.ts:34). ไม่ต้อง deploy ใหม่.
- prompt CMS ไม่ใช่ theater — นี่คือ headline: resolvePromptParts (prompt-resolver.ts:34-72) load PromptTemplate จาก DB ตอน inference จริง, DEFAULTS เป็นแค่ fallback. แก้ prompt ใน CMS เปลี่ยน string ที่ส่งเข้า Gemini จริง. KnowledgeDoc ก็ถูก retrieve + inject จริง (reading-service.ts:159-167).
- chart engine ของจริง 2,725 LOC + astronomy-engine, test ผ่าน, และ natal chart cache ไว้ (natal-chart-service.ts) ไม่ได้คำนวณใหม่ทุกข้อความ. requireReadyNatalChart hard-gate ป้องกันโมเดลมั่ว.
- API key ปลอดภัยกว่าหลาย startup ที่ระดมทุนแล้ว: เก็บแค่ชื่อ env var (secretReference) ไม่เคยเก็บค่า key ใน DB (schema.prisma:437, env.ts:69), และไม่มี SSRF surface เลยเพราะ AIProviderConfig ไม่มี baseUrl.
- Security ระดับ logic ทำได้ดีกว่าโค้ด pre-launch ทั่วไป: ไม่มี IDOR ใน money path, mass assignment ปิด (Zod strip + userId มาจาก session เสมอ), self-escalation ADMIN→SUPER_ADMIN บล็อก, banned user เผาเครดิตไม่ได้เพราะ reading-service.ts:82-86 re-read status จาก DB สด.
- สลิปไปที่ Vercel Blob ไม่ใช่ ephemeral serverless filesystem (proof/route.ts:39) — เป็นวิธีที่ payment flow บน Next.js ตายบ่อยที่สุด และคุณเลี่ยงได้. approval grant เป็น $transaction จริงพร้อม audit write. dead push subscription ถูก prune ตอน 410. typecheck ผ่าน, 73 tests ผ่าน.

## 🗺️ Roadmap (เรียงตาม leverage ไม่ใช่ตามมิติ)

### ต้องแก้ก่อนเปิดรับเงินจริง

- **[S] Config-hardening sprint: หมุนรหัส admin + verify email domain + ตั้ง Upstash + ตั้ง VAPID keys**  
  3 ใน 7 ขั้นของ business loop ของคุณพังที่ระดับ deploy ไม่ใช่ระดับโค้ด. รหัส admin ที่ published ยึด /admin ได้จาก repo สาธารณะ, email sandbox ทำให้ลูกค้าไม่รู้ว่าได้รับอนุมัติ + password reset ตาย, Upstash ที่ไม่ตั้งทำให้ rate limit เป็นภาพลวง. นี่คือความต่างระหว่าง "ระบบที่ลูกค้าซื้อจบ" กับ "ซื้อไม่จบ". ทั้งหมดเป็น config/wiring บนสถาปัตยกรรมที่ถูกต้องอยู่แล้ว.
- **[S] ปิด double-approve race ด้วย atomic compare-and-swap ใน reviewPayment**  
  สลิป 199฿ เดียวแจก 200 เครดิต + 2 subscription ได้เมื่อ admin กดซ้ำหรือ PATCH ถูก retry. เป็น money loss ตรง ๆ, fix แค่เปลี่ยน update เป็น updateMany({where:{status:PENDING}}) + throw เมื่อ count===0. เล็กแต่จำเป็นก่อนรับเงิน.
- **[S] แก้ ETA ที่เพี้ยน "199–2 วันทำการ" + เพิ่มเหตุผลปฏิเสธ/ETA/ช่องทางติดต่อบนหน้าจ่ายเงิน**  
  หน้าจอเดียวที่คนแปลกหน้าตัดสินใจโอน ฿199 แสดงข้อความที่อ่านแล้วเหมือน scam. fix เป็น string/UI ล้วน ใช้เวลาไม่กี่ชั่วโมง แต่กระทบ conversion โดยตรง.
- **[M] Validate proofUrl ให้เป็น blob ของเราเอง + เปลี่ยนสลิปเป็น private + rate-limit /payments/proof**  
  สลิปคือ PII ตาม PDPA ที่ตอนนี้เปิดสาธารณะถาวร, proofUrl แก้ได้ทำให้หลักฐาน approval อยู่ในมือ attacker, และ uploader ไม่มี rate limit เป็นช่อง DoS ค่า blob. สามเรื่องนี้อยู่รอบ money artifact ตัวจริงและเป็น fix ~10-30 บรรทัดต่ออัน.

### สัปดาห์แรกหลังเปิด

- **[M] สร้างหน้า usage ให้ user: /api/me/usage + "ใช้ไป 18/20 วันนี้" + แสดง balance บน chat header**  
  นี่คือครึ่งที่คุณอ้างว่า "เทียบเท่า ChatGPT" แต่ยังไม่ได้สร้าง. เป็น fix คุ้มที่สุด (ข้อมูลมีอยู่แล้วใน quota-service + app-data-provider) และเปลี่ยนประสบการณ์จาก "เจอ error แดงเป็นครั้งแรกที่รู้ว่ามี limit" เป็นเห็นล่วงหน้า. ลด ticket support ตั้งแต่สัปดาห์แรก.
- **[M] ปิดทางตันซื้อซ้ำ: เพิ่ม credit-only top-up product + แสดงฟอร์มให้ Pro user ที่ active**  
  loop รายได้ recurring พังตรงจุดที่ควรได้เงินซ้ำ — Pro user เครดิตหมดซื้อเพิ่มไม่ได้. banner สัญญา "เติมเครดิต" แล้วลิงก์ไปหน้าที่ไม่มีฟอร์ม. แก้ก่อนที่ลูกค้าที่ engage ที่สุด (คนจ่ายมากสุด) จะ rage-quit.
- **[M] ทำ quota enforcement ให้ atomic (re-check ใน tx + row lock/UsageCounter)**  
  limit ที่ admin config ทะลุได้ด้วย concurrency, คุณจ่าย Google เกิน cap ทุกวัน. คู่กับ P1 chart-drop หลัง 10 turns ที่หักเงินลูกค้าโดยไม่มีดวงในคำตอบ — สองเรื่องนี้คือ money leak ที่โตตามจำนวนลูกค้า active.
- **[S] แก้ RATE_LIMITED ที่ overload: แยก QUOTA_EXCEEDED ออก + ให้ client แสดง server message**  
  Pro user ที่ใช้ครบ monthly limit เห็น "ถามถี่เกินไป รอสักครู่" + ปุ่มลองใหม่ที่ไม่มีวันสำเร็จ — เป็นการโกหกลูกค้าที่จ่ายเงินว่าทำไมของที่จ่ายแล้วหยุดทำงาน, และไม่มีทางไปซื้อเพิ่มด้วย.

### เดือนแรก

- **[M] เพิ่ม Payment.packageId FK + ให้ grant derive ทุกอย่างจาก package row (ราคา, quota, expiresAt)**  
  การเปลี่ยน schema+service เดียวที่ leverage สูงสุด — ปิด amount/package decoupling, ให้ทุก payment มี audit trail ว่าซื้ออะไร, ทำ expiresAt=now+durationDays เป็นธรรมชาติ (ปิด never-lapse), และเป็นฐานของใบเสร็จ. เปลี่ยน payment flow จาก "admin เพ่งเลขแล้ว hardcode PRO" เป็น transaction ที่มีราคาสินค้าจริง.
- **[M] เพิ่ม cost tracking: price table ต่อโมเดล + เขียน estimatedCost + cap input tokens + spend-by-model บน dashboard**  
  ตอนนี้ operator cap token ได้แต่มองไม่เห็นต้นทุนเลย — estimatedCost เป็น NULL ตลอด, input token ไม่มี cap (ทุก KnowledgeDoc ยัดเข้า prompt ทุกครั้ง). นี่คือกลไกที่ Thai SaaS เงียบ ๆ ติดลบกระแสเงินสด. เปลี่ยน token log ที่ถูกต้องอยู่แล้วให้เป็น business intelligence จริง.
- **[L] เพิ่ม streaming (Gemini SSE / OpenAI stream) + cache transit chart บน Conversation**  
  ไม่มี streaming เลยคือช่องว่างใหญ่สุดเทียบกับ ChatGPT — user จ้อง spinner 10-30s. transit chart ถูก scrape myhora.com ใหม่ทุกข้อความ (natal cache แล้ว) เพิ่ม latency + เสี่ยงโดน IP ban. สองเรื่องนี้คือ perceived quality กับ fragility ที่โตตาม engagement.
- **[M] Observability + ops: Sentry, CI ที่รัน typecheck+lint+test, migrate step, after() สำหรับ notification**  
  ตอนนี้ approval ล้มตอนตี 2 = console.error เดียวแล้วเงียบ, ไม่มีอะไร page ใคร. notification ยิงเป็น void promise ที่ lambda ถูก freeze กลางคันได้. npm run lint แดง 8 errors (build "clean" ไม่จริง). ต้องมี safety net ก่อน scale.

### ทีหลังได้

- **[L] Admin: per-user throttle, top-spender view, payment reversal, admin viewer ของ readings/conversations, requireSuperAdmin gating, security headers, PDPA erasure**  
  เมื่อคุณรับ support person เข้ามา (ให้ role ADMIN) เขาแจกเครดิตตัวเองไม่จำกัดได้เพราะ requireSuperAdmin เป็น dead code. persona support agent ตรวจ ticket "reading พังแต่ถูกหักเงิน" ไม่ได้เลยเพราะไม่มี admin viewer. privacy policy สัญญาสิทธิ์ลบข้อมูลที่โค้ดทำไม่ได้ (PDPA §33). เป็นเรื่องของ operational maturity ไม่ block first sale.

### 3. `P1` `S` Rate limit ใช้ไม่ได้จริงบน production — Upstash ไม่ได้ตั้ง limit เลย fail-open เป็น per-lambda memory

### 4. `P1` `S` Double-approve race แจกเครดิตซ้ำ + สร้าง subscription 2 อัน (ไม่มี status guard ใน transaction)

**ความเสียหายทางธุรกิจ:** admin สองคนกดอนุมัติสลิปเดียวกัน (หรือ PATCH ถูก retry บนเน็ตมือถือกระตุก). ทั้งคู่อ่าน status=PENDING นอก tx และผ่าน check. tx.payment.update ไม่มี predicate บน status — ภายใต้ READ COMMITTED update ตัวที่สอง block บน row lock แล้ว apply ต่ออยู่ดี (ไม่มี filter ให้ re-evaluate จึงไม่มีทาง 0 rows). ผลจากสลิป 199฿ เดียว: 200 เครดิตแทน 100, 2 subscription, 2 audit. addCredits ฝั่ง grant เป็น bare increment ไม่มี version guard.

**ไฟล์:** `src/server/payment/payment-service.ts:141`

**วิธีแก้:** เปลี่ยน tx.payment.update เป็น guarded write: `updateMany({ where: { id, status: "PENDING" }, ... })` แล้วถ้า count===0 throw AppError เพื่อ rollback ทั้ง tx ก่อนถึง addCredits/subscription. เพิ่ม unique constraint บน CreditTransaction(referenceType, referenceId, type) กัน duplicate grant.

| **หมวด** | concurrency |
| **พบโดย** | Money loop |
| **ผลโหวต** | ยืนยัน 3/3 |

**หลักฐานในโค้ด:**


**เกิดอะไรขึ้นจริง:** Two admins tap the same Web Push notification and both hit อนุมัติ (or one PATCH is retried on a flaky mobile connection). ## 📐 รายละเอียดรายมิติ (8 มิติ)

### Money loop: สลิป → อนุมัติ → เข้าถึง — **4/10**

The happy path is real and better-built than most manual-payment MVPs: the grant is one `$transaction`, `addCredits` genuinely honors the passed tx client, every review is audited, and both sides get notified. But three things make me refuse to call it production-ready for money. (1) The approve path has a real, reachable double-grant: `reviewPayment` checks `status==='PENDING'` OUTSIDE the transaction and then issues `tx.payment.update({where:{id}})` with no status predicate — under Postgres READ COMMITTED the second concurrent UPDATE blocks on the row lock, re-reads the now-APPROVED row, and applies anyway (no filter to re-evaluate, so no zero-row outcome), producing two subscriptions and double credits for one 199฿ slip. (2) What the user pays is completely decoupled from what they get: `amount` is a free-text number from the client, the admin panel hardcodes `packageCode:"PRO"`, the service defaults to `"PRO"`, and nothing anywhere compares `payment.amount` to `Package.price` — the panel doesn't even display the price so the admin could compare. Payment has no packageId column, so there is no record of what was bought. (3) There is no unwind. An APPROVED payment is terminal, and the only clawback tool (`adjustUserCredits` → `deductCredits`) throws NO_QUOTA once the user has spent the credits — a fake slip that burns its credits is unrecoverable without psql. Add a permanently-stuck PENDING that hard-blocks the customer from ever resubmitting, and this is demo-grade: it will work fine for the first 20 honest customers and hurt on the first dishonest one or the first double-click.

**จุดแข็งในมิตินี้:**
- The grant really is atomic, and `addCredits` really is tx-aware — I checked rather than assumed. `credit-service.ts:107-109` does `"$transaction" in client ? client.$transaction(run) : run(client)`; a `Prisma.TransactionClient` has no `$transaction` member, so passing `tx` from `payment-service.ts:187` runs the wallet update + ledger row INSIDE the caller's transaction. Payment status, subscription cancel/create, credit grant and `writeAudit` all commit or roll back together (payment-service.ts:140-206).
- `deductCredits` uses a genuine optimistic lock: `updateMany({where:{userId, version: wallet.version}})` and throws if `count === 0` (credit-service.ts:51-59). The spend side of the wallet is race-safe even though the grant side is not.
- Append-only ledger discipline: every balance change writes a `CreditTransaction` row with type/referenceType/referenceId, and the payment grant references the payment id (payment-service.ts:180-186), so a human can reconstruct why a balance moved.
- The duplicate-PENDING guard is enforced server-side (payment-service.ts:25-30) AND reflected honestly in the user UI, which swaps the form for a 'รอแอดมินตรวจสอบ' card showing the submitted slip (payment-submit-card.tsx:109, 134-161). Users can't spam the admin queue.
- Slip upload validates MIME allowlist and a 2MB cap server-side, and scopes the blob path per user (proof/route.ts:6-7, 29-34, 38).
- The reject reason is not silently swallowed — `notifyUserPaymentReviewed` emails the admin's note to the user (payment-service.ts:208-213, payment-notify.ts:64-80).
- Real tests exist for the branches that matter: duplicate-pending rejection, missing proofUrl, approve-grants-credits (asserting `addCredits` is called WITH the tx object), and reject-adds-no-credits (tests/payment-service.test.ts:60-155).

### Usage metering: จริง ถูกต้อง user เห็นไหม — **5/10**

The *skeleton* of the metering system is genuinely well designed — better than most solo projects. Tokens are REAL provider counts (not chars/4), every user-facing AI call goes through one funnel (createReading), failures/timeouts are logged with status FAILED/TIMEOUT and are NOT charged, and the charge + reading + usage log commit in a single $transaction guarded by DB-level idempotency and an optimistic wallet lock. The classic metering bug (charge-before-call, no refund) is NOT present. That earns real credit.

But the owner's specific claim — "usage is really tracked, users can see it, admin can set tokens — on par with ChatGPT" — is half true and half false, and the false half costs money:
(a) COST IS NEVER RECORDED. There is no price table anywhere in the repo. `estimatedCost` is accepted by logUsage but never passed by either call site, so the column is always NULL and the admin's own usage table renders an always-empty "cost" column. The operator literally cannot answer "what did I spend on Gemini this month" from their own product. Combined with unbounded input tokens (every enabled KnowledgeDoc + full thread history injected into every call, only the OUTPUT is capped), this is how a Thai SaaS quietly goes cash-flow negative.
(b) THE USER SEES ALMOST NOTHING. One number — credit balance — on /account. No usage history, no "3/20 calls used today", no ledger. The daily/monthly limits the admin configures are fetched from the DB and then thrown away before rendering. The user's first contact with their quota is a red error toast. That is not on par with ChatGPT.
(c) THE QUOTA IS RACY. The daily/monthly limit is a plain COUNT before the AI call, with no re-check inside the transaction — despite a code comment claiming otherwise. Ten parallel requests (the route rate limit allows 10/min) all pass a check the admin thinks is a hard cap.
(d) Two AI call sites (admin config test, admin health probe across all enabled configs) burn real provider tokens and write ZERO AIUsageLog rows.

5/10: the ledger is honest about what it records, but it records no money, enforces its limits on the honor system under concurrency, and shows the customer nothing. Ship-blocking for a paid product? Not quite. Week-one-regret? Yes.

**จุดแข็งในมิตินี้:**
- Charge-AFTER-success is implemented correctly — the single most important thing to get right, and it is right. On AI failure/timeout, usage is logged with status FAILED/TIMEOUT and NO credit is deducted (src/server/horoscope/reading-service.ts:194-207); the error is surfaced as a typed AppError and the Thai UI even tells the user 'ไม่ถูกหักเครดิต' (src/components/app/chat-view.tsx:56-57). No refund reconciliation is needed because no speculative charge ever happens.
- Credit deduction + reading row + usage log commit in ONE prisma.$transaction (src/server/horoscope/reading-service.ts:210-251). You cannot get a charge without a usage log, or a usage log without a charge. This is the correct shape.
- Tokens are REAL provider-reported counts, not a chars/4 estimate: src/server/ai/providers/gemini.ts:131-134 reads usageMetadata.promptTokenCount / candidatesTokenCount, and src/server/ai/providers/openai.ts:105-108 reads usage.prompt_tokens / completion_tokens. Genuinely trustworthy token data.
- No real streaming means no streaming-abort metering hole. The 'stream' at src/components/app/chat-view.tsx:245-279 is a client-side typewriter over a fully-buffered server response — so the provider usage block can never be lost mid-stream. (Worth knowing this is why that whole class of bug is absent.)
- Bangkok boundary math is correct and stable. BKK_OFFSET_MS = 7h (src/server/credit/quota-service.ts:11) — Thailand is ICT/UTC+7 year-round and has observed no DST since 1952, so this constant will not drift. The day/month boundary computation (quota-service.ts:13-22) correctly converts a Bangkok calendar date back to a UTC instant.
- Idempotency is enforced at the DATABASE, not just in app code: @@unique([userId, idempotencyKey]) on horoscope_readings (prisma/schema.prisma:513) plus the early-return at reading-service.ts:67-79. A double-click or a retry with the same key genuinely cannot double-charge.
- The wallet uses an optimistic-lock version bump (src/server/credit/credit-service.ts:51-59) inside the transaction, so concurrent deductions cannot overdraw the balance — the credit half of the quota system IS race-safe (unlike the call-count half).
- Every user-facing AI entry point funnels through createReading — /api/horoscope/readings/route.ts:37 and /api/conversations/[id]/messages/route.ts:26 → message-service.ts:75. The application code is genuinely competent — typecheck is clean, all 73 tests pass, the approval grant is a real Prisma $transaction with an audit write, slips go to Vercel Blob (not local disk, so serverless is fine), and dead push subscriptions are pruned on 410. That is not the problem. The problem is that the *deployed* system does not work. Three of the seven steps in the owner's own business loop are broken in production right now: (3) the admin never gets a push notification because VAPID keys exist in no env file anywhere; (4→5) the customer is never told they were approved because EMAIL_FROM is Resend's sandbox sender that only delivers to the account owner. This is not "one week of polish"; it is a system that a real customer cannot complete a purchase through today. A 3, not lower, because every one of these is a configuration/wiring fix on top of an architecture that is basically right — a focused two days gets this to a 7.

### Security ของ money + AI surface — **5/10**

The application-logic security here is genuinely better than most pre-launch code: no IDOR anywhere in the money paths, mass assignment is closed (Zod strips unknown keys and userId always comes from the session), the ADMIN→SUPER_ADMIN self-escalation path is explicitly blocked, no SSRF surface exists (AIProviderConfig has no baseUrl at all), API keys are stored as env-var *names* not values, audit rows are session-attributed and have no delete route, and — contrary to my initial hypothesis — a banned user genuinely CANNOT burn AI credits, because reading-service.ts:82-86 re-reads status from the DB and both the reading route and the chat route funnel through it. That is real engineering, not luck.

But the score is capped by two things a staff engineer would not ship. (1) Every Thai bank slip — payer's real name, account number, amount — is uploaded to a PUBLIC Vercel Blob at a fully deterministic path (`payment-slips/<userId>/<epochMs>.jpg`, no random suffix in @vercel/blob 2.6.1) and served forever with no authentication. That is PDPA-regulated personal data on an open URL. (2) `proofUrl` is accepted as ANY user-supplied URL — the system never verifies the slip it stores is the slip it uploaded, so the evidence trail behind every approval is attacker-mutable. Both are ~10-line fixes, which is why this is a 5 and not a 3: the architecture is sound, the perimeter around the actual money artifact is not. Separately, JWT role/status is baked at sign-in and never revalidated for admins, so there is no way to revoke a compromised or fired admin short of rotating AUTH_SECRET.

### Data integrity & correctness at scale — **7/10**

The money-critical core — the credit wallet + ledger — is genuinely production-grade: atomic decrement guarded by an optimistic version lock, an immutable CreditTransaction written in the same DB transaction, idempotency keys on readings/messages, "no charge on AI failure," and tx-aware writeAudit/addCredits helpers. Catalog/user/payment services correctly wrap multi-table mutations in $transaction, and the Bangkok timezone math is consistent between quota and dashboard (and correct, since Thailand has no DST). That is well above demo-grade. It is NOT a 9-10 because there is one real money bug (payment approval is a check-then-write with no atomic compare-and-swap, so concurrent/duplicate approvals double-grant credits and Pro), and a consistent seam where the CMS content layer (FAQ/prompt/knowledge/settings publish + drafts) abandons the transactional discipline used everywhere else — revisions and audit rows are written non-atomically, revision version numbers race with no unique constraint, and drafts have no optimistic locking so co-admins silently clobber each other. Add offset-pagination + unindexed ILIKE search that degrade at 100k–1M rows, and the honest verdict is: the paid-usage loop is solid, but the approval path and the admin CMS have correctness gaps a real operator will eventually hit.

**จุดแข็งในมิตินี้:**
- Credit ledger is the strongest part of the system: deductCredits (credit-service.ts:38-74) does a balance precheck, then an atomic updateMany guarded by `version` (line 51-54) that decrements balance and bumps version in one statement, throws on lost-update (line 57), and appends an immutable CreditTransaction in the SAME tx — the wallet and ledger cannot diverge as long as every mutation goes through this service, and nothing writes balance directly.

## What the audit missed

### 1. `npm run lint` is RED — 8 errors. The audit ran typecheck + test and declared the build "clean," but never ran lint.
The project's own documented pre-commit gate is `npm run typecheck && npm run lint && npm run test` (BACKEND_TASKS.md:61, FRONTEND_TASKS.md:63). I ran all three: typecheck passes (exit 0), 73 tests pass (exit 0), but `npm run lint` exits non-zero with **8 errors, 4 warnings**. The two hard errors are React-19 `react-hooks/set-state-in-effect` violations in load-bearing components:
- `src/components/theme-provider.tsx:42` — `setThemeState(initial)` synchronously in an effect. This wraps the entire app's theming.
- `src/components/app/nav-progress.tsx:20` — `setVisible(false)` synchronously in an effect.

These are the new React Compiler-era lint rules flagging cascading-render bugs. `next build` in Next 16 no longer runs ESLint by default, so this won't block the Vercel build — but the team's stated quality gate is failing, and "the build is clean" is not true. This is the single most embarrassing miss because it's one command.

### 2. PDPA right-to-erasure is a written promise the product cannot technically keep.
The published privacy policy and terms (defaults in `src/lib/cms-keys.ts:184`, `:186`, `:258`) explicitly promise users: *"ขอเข้าถึง แก้ไข หรือลบข้อมูลส่วนบุคคลของคุณได้"* and *"หากต้องการลบบัญชี… การลบบัญชีจะเป็นการลบข้อมูลถาวร"* (you may request access/edit/**deletion** of your personal data; account deletion is permanent). But there is **zero deletion capability anywhere in the codebase**: `grep` for `user.delete`/`deleteMany` on the User model returns nothing (only token and push-subscription cleanup). There is no self-serve delete, and the admin API is GET-only for users — `src/app/api/admin/users/route.ts` and `src/app/api/admin/users/[id]/route.ts` export only `GET`; the DELETE-route inventory contains no user route. Combined with the audit's finding that bank slips live at permanent public blob URLs (also with no deletion path), a Thai business subject to PDPA has made a §33 erasure promise it can only honor via `psql`. This is legal/compliance exposure the persona reviewers didn't surface.

### 3. The Resend sandbox-sender problem is far wider than "the approval email fails."
The audit scoped `EMAIL_FROM` (default `onboarding@resend.dev`, which only delivers to the Resend account owner — `src/server/email/mailer.ts:29`) to payment approval/rejection. But the **same** `sendEmail` is the transport for:
- **Email verification on register** — `src/app/api/auth/register/route.ts:41` → `sendVerificationEmail`.
- **Password reset** — `password-reset-service.ts`.

So in production the blast radius is **account recovery is dead**: any real user who forgets their password gets a reset link that vanishes into the sandbox void, and the verification banner's "resend" button silently no-ops forever. Mitigating nuance (which lowers severity of one part): email verification is a *soft banner only* — `needsEmailVerification` (email-verification-service.ts:88-94) drives UI, nothing gates app usage on it — so unverified users can still use the product. The password-reset lockout is the real new harm.

### 4. No receipt / tax invoice exists at all — for a business collecting THB.
`grep` for `receipt|ใบเสร็จ|ใบกำกับ|tax.invoice` across `src/` returns nothing. A Thai customer who pays 199฿ gets no receipt document of any kind, and the operator has no record they can hand to their own accountant. Payment has no `packageId` and no receipt artifact, so the financial paper trail is: a free-text `amount` and a mutable `proofUrl`. For a money-taking business this is both a customer-trust and a bookkeeping gap.

### 5. There is no scheduled-execution capability in the entire system.
No `vercel.json`, no cron config, no queue/worker, and `next.config.ts` is the empty stub (`/* config options here */` — which also confirms the audit's "no security headers"). The consequence the zoomed-in reviewers missed: **nothing time-based can ever run.** There is no sweep to enforce PDPA slip retention/deletion, no retry for the emails that silently fail through the sandbox sender, and no reconciliation for a payment stuck in PENDING. On Vercel, cron is nearly free (`vercel.json` `crons`), and its total absence is why the "money/ops half" of the product has no safety nets.

### 6. Correction for the record: "Pro never lapses" is NOT a missing-expiry-logic problem.
The audit lists this as P2 and implies the expiry story is broken. In fact the access path **does** honor expiry, lazily, at read time: `account-service.ts:13` and `:40` and `quota-service.ts:29` all filter subscriptions with `OR: [{ expiresAt: null }, { expiresAt: { gt: now } }]`. So no cron is needed to expire access — the moment `expiresAt` passes, the plan reads as FREE. The entire bug is that the grant in `payment-service.ts` writes `expiresAt: null`. The fix is one line at grant time (set `expiresAt = now + 30d`), not an expiry subsystem. Worth correcting so the owner doesn't over-scope it.

### 7. Next 16 idiom point the completeness brief asked for: admin publish never invalidates a cache, and marketing pages are all `force-dynamic`.
Every public/marketing page opts out of static generation — `src/app/(public)/{page,pricing,privacy,terms,disclaimer,help,contact}.tsx` all set `export const dynamic = "force-dynamic"`. So the marketing site (rebuilt in the recent "revamp public marketing site" commit) hits Postgres on every anonymous request instead of being statically cached/ISR'd. Meanwhile the CMS-content caches (`settings-service.ts:182` consent texts, `category-service.ts:31` public categories, `session-guard.ts:63` maintenance mode) use **time-based `unstable_cache({ revalidate: 45..60 })` with no cache tags**, and there is **no `revalidateTag`/`revalidatePath` call anywhere in `src/`**. Net effect: an admin publish (privacy text, categories, maintenance toggle) takes up to 45–60s to appear and cannot be pushed immediately, and the money-losing combination — dynamic-render on every hit yet still stale for a minute after an edit — is the worst of both. The idiomatic Next-16 fix (static/ISR marketing pages + `revalidateTag` on publish) is untapped.

### The single highest-leverage architectural change
Introduce a **`Payment.packageId` foreign key and make the grant derive everything from the purchased package row**, replacing the free-text `amount` + hardcoded `packageCode:"PRO"` + null-expiry grant. This one change simultaneously closes the audit's biggest money defects and several of mine: it gives every payment an auditable link to what was actually bought (fixes the missing paper trail / receipt basis), lets the review screen compare `amount` vs `Package.price` (fixes the "pays 199 gets PRO regardless" decoupling), and makes `expiresAt = now + package.durationDays` a natural part of the grant (fixes never-lapse). It is a schema-plus-one-service-function change that converts the payment flow from "admin eyeballs a number and hardcodes PRO" into a real product-priced transaction — higher leverage than any of the individual UI/rate-limit fixes because it repairs the data model the whole money loop is missing.

*(Minor note, not a lead finding: the fortune-telling "เพื่อความบันเทิง" legal disclaimer lives only on the footer `/disclaimer` page and in CMS defaults; the chat surface shows only a generic AI-accuracy line — `chat-view.tsx:769` "อาจให้ข้อมูลที่ไม่ถูกต้อง… โปรดใช้วิจารณญาณ" — not the entertainment-only disclaimer at the point the fortune is delivered.)*
