## 1. บริบทและเป้าหมาย

จากข้อเสนอของลูกค้าเรื่องการควบคุมต้นทุน Gemini ต่อผู้ใช้ต่อเดือน ทีมพัฒนาได้ปรับระบบแชทให้ **ลดโทเคน input ต่อเทิร์น** โดยเฉพาะเทิร์นถัดไป (follow-up) ซึ่งเป็นจุดที่เผาโทเคนมากที่สุด

**เป้าหมาย:** ลด input ต่อรอบสนทนาจากระดับหลักหมื่นโทเคน ให้ใกล้เคียงสมมติฐาน ~2,000–4,000 โทเคน/รอบ (ตามชีตที่ลูกค้าคำนวณ) และช่วยให้ต้นทุนรวมต่อคนต่อเดือนเข้าใกล้เป้า ~111k โทเคน/คน/เดือน

**หลักการสำคัญที่ยังคงไว้:** ระบบยังเป็น **engine-first** — ทุกคำตอบยังอ้างอิงจากตาราง `[natal]` และ `[memory]` ที่คำนวณจาก engine ห้าม AI แต่งตำแหน่งดาวเอง

---

## 2. ปัญหาที่พบก่อนแก้ไข

ทุกเทิร์นแชท ระบบส่งข้อมูลเข้า Gemini แบบเต็ม ๆ ดังนี้:

| ส่วนที่ส่ง | สถานะก่อนแก้ | ผลกระทบต่อโทเคน |
|-----------|--------------|-----------------|
| ประวัติแชท | สูงสุด **10 รอบ** คำตอบ AI เต็มทุกข้อความ | สูงมาก (~1,650 tok/คำตอบ) |
| ตารางดวง `[natal]` | ตารางสมผุส/ทักษาเต็ม **ทุกเทิร์น** | ~1,000–2,000 tok/เทิร์น |
| `[memory]` | ทุกหมวด (งาน รัก เงิน สุขภาพ) | ซ้ำซ้อนทุกเทิร์น |
| Knowledge docs | เอกสารที่เปิดใช้งาน **ทั้งหมด** ไม่จำกัดความยาว | สะสมใน system prompt |
| Output | `maxOutputTokens = 2,048` **เท่ากันทุกแพ็กเกจ** | output แพงกว่า input 3–6 เท่า |

## 3. สิ่งที่ปรับแล้ว (Phase 1–3)

### Phase 1 — จำกัดประวัติแชท

| รายการ | ก่อน | หลัง |
|--------|------|------|
| จำนวนรอบที่ส่งให้ AI | 10 รอบ | **4 รอบ** |
| คำตอบ AI ในประวัติ | เต็มทุกข้อความ | ตัดเหลือ **600 ตัวอักษร** + `…` |
| ดึงข้อความจาก DB | ทั้งเธรด | **12 แถวล่าสุด** เท่านั้น |

คำถามของผู้ใช้ในประวัติยังเก็บ **เต็ม** — ตัดเฉพาะคำตอบ AI เก่าเพื่อประหยัดโทเคน

### Phase 2 — ดวง natal (สถานะจริงในโค้ด)

| สถานการณ์ | บล็อกดวงที่ส่ง |
|-----------|---------------|
| ทุกเทิร์น (รวมเทิร์นแรก) | บล็อก `[natal]` **แบบย่อ** (`useCompactNatal = true` ใน `prompt-builder.ts`) |
| โหมดดวงจร (TRANSIT) | ตาราง **เต็ม** เหมือนเดิม |

> เอกสารเดิมเคยวางแผนว่าเทิร์นแรกใช้ตารางเต็ม — implementation ปัจจุบันใช้ compact natal ทุกเทิร์นเพื่อประหยัด token; ถ้าต้องการกลับไปเต็มเฉพาะเทิร์นแรก ต้องแก้ `prompt-builder` แยกงาน

**`[memory]`:** ส่งเฉพาะหมวดที่ผู้ใช้ถาม (เช่น ถามเรื่องความรัก → ส่ง memory หมวด love เท่านั้น)

### Phase 3 — จำกัด Knowledge และ Output

| รายการ | ค่าใหม่ |
|--------|---------|
| Knowledge docs รวม | สูงสุด **4,000 ตัวอักษร** (เรียงตามลำดับความสำคัญ ตัด doc ท้ายออกเมื่อเกิน) |
| Output tokens — Free | สูงสุด **1,024** |
| Output tokens — Pro | สูงสุด **2,048** |

ค่า output จะไม่เกินเพดาน Admin ที่ตั้งไว้ — ใช้ค่าที่ **ต่ำกว่า** ระหว่าง Admin config กับเพดานแพ็กเกจ

## 4. ประมาณผลลัพธ์ต่อเทิร์น (follow-up)

| ส่วน | ลดลงโดยประมาณ |
|------|----------------|
| ประวัติแชท | **70–80%** |
| บล็อกดวง | **60–70%** |
| memory (เฉพาะหมวด) | ลดตามจำนวนหมวดที่ไม่ส่ง |
| knowledge | จำกัดที่ 4,000 ตัวอักษร |
| output (Free) | ครึ่งหนึ่งของเดิม |

**รวม input ต่อเทิร์น follow-up:** จากหลักหมื่นโทเคน → ประมาณ **2,000–4,000 โทเคน** (ใกล้สมมติฐาน 5k/รอบในชีตลูกค้า)

---

## 5. สิ่งที่ **ไม่ได้เปลี่ยน** (เพื่อคุณภาพคำตอบ)

- กฎ **engine-first** — AI ต้องอ้างอิง `[natal]` / `[memory]` / `[transit]` เท่านั้น
- เทิร์นแรกยังได้ตารางดวงเต็มสำหรับความแม่นยำ
- โหมดดวงจร (TRANSIT) ยังได้ตารางเต็ม
- การคิดเครดิต / quota / idempotency ทำงานเหมือนเดิม
- คุณภาพการแสดงผลแชท (Markdown, streaming) ไม่ได้ถูกตัดออก

---

## 6. วิธีตรวจสอบหลัง deploy

1. เข้า **Admin → AI Usage** (หรือหน้า log การใช้ AI)
2. ทดสอบแชท:
   - เทิร์นแรก 1 ครั้ง
   - follow-up ต่อในเธรดเดิม 2–3 ครั้ง
3. เปรียบเทียบค่า **`inputTokens`** และ **`outputTokens`** ก่อน/หลัง deploy
4. ตรวจว่าคำตอบ follow-up ยังอ้างอิงดาว/เรือนถูกต้องและตอบตรงคำถาม

> ระบบบันทึก `inputTokens` / `outputTokens` จาก Gemini ทุกครั้งที่เรียก AI อยู่แล้ว — ไม่ต้องติดตั้งเพิ่ม

---

## 7. สิ่งที่ยังไม่ทำในรอบนี้ (Phase 4 — วางแผนไว้ภายหลัง)

หากแชทยาวเป็นพฤติกรรมปกติของผู้ใช้ อาจเพิ่มในอนาคต:

- คอลัมน์ `Conversation.summary` ในฐานข้อมูล
- สรุปเธรดเก่าด้วย AI (Flash-Lite) เมื่อประวัติเกิน 5 รอบ
- ส่ง: summary + 3 รอบล่าสุด + memory

**รอบนี้ยังไม่จำเป็น** — Phase 1–3 คาดว่าพอเข้าเป้าต้นทุนที่ลูกค้าเสนอแล้ว

## 9. สรุปสั้น ๆ (Executive Summary)

> ปรับระบบแชทให้ส่งข้อมูลน้อยลงในเทิร์นถัดไป — จำกัดประวัติ 4 รอบ, ย่อตารางดวง, ส่ง memory เฉพาะหมวด, จำกัด knowledge และ output ตามแพ็กเกจ Free/Pro — โดยยังคงกฎ engine-first และคุณภาพคำตอบเทิร์นแรก/ดวงจรไว้ครบ

หากต้องการปรับตัวเลข (เช่น 4 รอบ → 5 รอบ, หรือ output Free 512) แจ้งทีมพัฒนาได้ — ค่าทั้งหมดรวมศูนย์ที่ `src/config/constants.ts` ปรับได้โดยไม่ต้องแก้ logic หลัก
